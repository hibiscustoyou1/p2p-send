# 📝 产品文案与沟通规范 (Product Guidelines)

虽然这是一套强大的基础设施 Template，但在最终展现给用户与内部开发对接时，我们需要保持高度的精准和专业性。

## 1. 结构化任务与 AI 沟通流 (Structural Communication)
所有系统内的记录对话与计划汇报，必须采用标准的分块模型保障开发轨迹透明化：
- **`[架构师规划]`**：针对目前 Context 的溯源过程，以及准备使用的解决思路。
- **`[当前任务]`**：指出本轮即将变更的特性。
- **`[执行动作]`**：描述代码的精准动作点。
- **`[校验结果]`**：预期的前后端表现或者终端校验结果。
- **中文语言覆盖**：所有非代码块输出内容，都必须 100% 为简体中文。

## 2. 前置界面反馈与客户端提示文案
- **直指人心**：对于操作状态的通知，必须简洁有力。使用“保存成功”、 “网络超时，请重试” 之类的标准化用语。
- **优雅的异常拦截**：绝对禁止把服务端带有系统表名的 Raw Error (如 SQL 语法错误、500 Call Stack) 直接反馈到用户前端。必须设立 API 层面的错误字典与全局捕获守卫，输出业务级别的友好提醒。
